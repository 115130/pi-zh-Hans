收尾。

附加说明：$ARGUMENTS

首先根据对话历史确定上下文。

上下文检测规则：
- 如果对话已提及某个 GitHub issue 或 PR，使用该现有上下文。
- 如果工作来自 `/is` 或 `/pr`，则认为 issue 或 PR 上下文已从对话和已完成的分析工作中明确。
- 如果对话历史中没有 GitHub issue 或 PR，则视为非 GitHub 工作。

除非我在本次请求中明确覆盖了某项内容，否则按顺序执行以下操作：

1. 根据仓库的变更日志规则，在相关包的 `## [Unreleased]` 下添加或更新变更日志条目。
2. 如果此任务与某个 GitHub issue 或 PR 关联，且本次会话中尚未发布最终的 issue 或 PR 评论，则以我的语气起草评论，预览后，发布恰好一条最终评论。评论必须以此确切独立的声明行结尾，不得有任何变体：

   ```text
   This comment is AI-generated by `/wr`
   ```
3. 仅提交你在此会话中更改的文件。
4. 如果此任务恰好关联一个 GitHub issue，在提交信息中包含 `closes #<issue>`。如果关联多个 issue，停止并询问使用哪一个。如果未关联任何 issue，不要在提交信息中包含 `closes #` 或 `fixes #`。
5. 检查当前 git 分支。如果不是 `main`，停止并询问如何处理。除非我明确说明，否则不要从其他分支推送。
6. 推送当前分支。

约束：
- 永远不要暂存不相关的文件。
- 永远不要使用 `git add .` 或 `git add -A`。
- 如果代码有变更，在提交前运行必要的检查。
- 除非我明确要求，否则不要创建 PR。
- 如果这不是 GitHub issue 或 PR 的工作，不要发布 GitHub 评论。
- 如果本次会话中已发布过最终的 issue 或 PR 评论，除非我明确要求，否则不要再发布。
