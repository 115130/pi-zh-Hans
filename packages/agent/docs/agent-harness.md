# AgentHarness 生命周期

`AgentHarness` 是底层代理循环之上的编排层。它拥有会话持久化、运行时配置、资源解析、操作锁定以及面向扩展的变更语义。

本文档描述了当前方向和已实现的行为。某些扩展/会话门面细节已计划好并明确标出。

## 最终的生命周期目标

Harness 监听器和钩子应该能够捕获 `AgentHarness` 实例，并从中档 API 被记录为允许的任何事件中调用公共 harness API。这些调用不得损坏正在进行的回合快照、重新排序已持久化的记录条目、丢失待处理的写入、使结算死锁，或将 harness 留在错误的阶段。

预期的规则是：

- 忙时结构操作仍然被拒绝
- 队列操作在文档中记录的安全回合点被接受
- 运行时配置设置器更新未来快照，而不改变当前的提供商请求
- 忙时的会话写入被持久化排队并按确定性顺序刷新
- 获取器返回最新 harness 配置，而不是正在进行的快照
- 监听器/钩子当前不接收任何门面；如果它们捕获原始 harness 并在活动运行期间调用诸如 `waitForIdle()` 之类的结算 API，可能会死锁。未来的门面应该暴露 `runWhenIdle()` 代替。

`AssistantMessageStream` 已经将提供商传输流（如 SSE 或 WebSocket 读取）与下游事件消费解耦。因此 harness 可以等待监听器、扩展钩子、持久化和保存点工作，而不会阻塞提供商传输读取器或重新引入临时事件队列。生命周期代码应优先在 harness 边界使用显式的带等待的序列化，而不是即发即弃的钩子/事件结算。

最终的生命周期加固过程应通过广泛的监听器/钩子重入测试套件来证明这些保证。

## 错误处理

当前的划分是：

- 底层能力和辅助函数使用 `Result<TValue, TError>`，其中预期的失败被包含且不得抛出，例如 `ExecutionEnv`、文件系统/shell 操作、shell 输出捕获、资源加载和压缩辅助函数
- 高级变更/编排 API（如 `Session` 和 `AgentHarness`）拒绝/抛出而不是返回可以被忽略的裸结果
- 公共 `AgentHarness` 失败在可行时规范化为 `AgentHarnessError`；子系统错误作为 `cause` 保留

## 会话持久化

会话作为仅追加的 JSONL 文件持久化。每条记录是一个独立的 JSON 对象，代表一个状态变更：

```typescript
interface SessionEntry {
  id: string;           // 唯一 ID
  parentId?: string;    // 父条目 ID（用于树结构）
  timestamp: string;    // ISO 8601 时间戳
  type: string;         // 条目类型
  data: unknown;        // 类型特定的数据
}
```

## 操作锁定

Harness 在关键部分使用操作锁来防止并发修改：

- `busy` 标志防止在代理运行时进行结构性更改
- 队列操作在代理忙时累积，并按顺序处理
- 会话写入使用写锁确保持久化顺序

## 资源解析

资源在会话创建时解析，并在整个会话生命周期中缓存：

- 扩展在会话启动时加载
- 技能已解析并可访问
- 提示模板已解析并可访问
- 主题已解析并可访问
- 上下文文件已发现并可访问

## 扩展状态变更

扩展可以通过 `pi.appendEntry()` 将会话条目写入会话，在重启后仍然存在。扩展还可以通过 `pi.state` 设置和获取内存中的键值状态。
