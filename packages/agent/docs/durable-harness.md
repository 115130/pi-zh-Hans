持久化 AgentHarness / 会话设计笔记。

## 框架

完全持久化的 `AgentHarness` 本身是不现实的，因为重要的依赖是宿主应用提供的运行时 JS：

- 工具实现
- 模型/认证提供商
- 扩展和钩子处理器
- 资源加载器
- 系统提示回调/修饰器

实际目标是半持久化的 harness：

- 会话是持久化的仅追加状态树
- harness 将其拥有的状态持久化到会话条目中
- 宿主应用负责在恢复时重建兼容的非可持久化依赖
- 恢复从持久化边界重新开始，而不是从正在进行的提供商流中

## 会话拥有持久化状态

将会话视为所有持久化代理状态，而不仅仅是对话记录。

现有的会话状态已包含 harness 状态：

- 模型变更
- 思考级别变更
- 压缩
- 分支摘要
- 标签

## 半持久化恢复流程

恢复时：

1. 从会话文件重建消息历史
2. 宿主重新注册模型、工具、扩展、钩子
3. harness 从最后已知状态恢复（如果有）
4. 代理从停止处继续

## 持久化边界

持久化边界是 harness 保证已将状态写入会话条目的点：

- 每条消息后
- 每次压缩后
- 每次模型变更后
- 每次思考级别变更后
- 每次工具执行后

在这些边界上，可以安全地关闭和恢复 harness 而不会丢失状态。

## 限制

- 正在进行的 LLM 流在恢复时会丢失（从最后一条持久化消息重新开始）
- 宿主必须提供与创建时相同的依赖
- 扩展状态不在会话中持久化（除非扩展使用 `pi.appendEntry()`）
